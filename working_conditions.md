Практическая работа №1. Кодовая база и зависимости
Требуемые условия завершения
Цель: научиться отделять код от конфигурации и соблюдать принцип "один репозиторий для одного приложения".
Шаг 1: Организация репозиториев
Создайте новое простое веб-приложение (например, на Python/FastAPi, Node.js/Express или любом другом фреймворке).
Разбейте его на логические модули (роутинг, работа с БД, бизнес-логика).
Убедитесь, что в репозитории нет жестко прописанных путей к файлам, специфичных для вашего локального компьютера (например, C:\Users\... или /home/user/...).
Шаг 2: Инициализация репозитория
Инициализируйте Git-репозиторий и сделайте первый коммит. Создайте две ветки (staging/prod), чтобы понять, что ветки — это часть одной кодовой базы, а не разные приложения.


Практическая работа №2. Полная изоляция с помощью Docker

Цель: Запаковать приложение в контейнер, чтобы оно гарантированно работало одинаково на компьютере разработчика, на сервере тестирования и в продакшене, независимо от того, какие версии Python/Node.js или библиотек установлены в основной системе.

Шаг 1: Подготовка приложения
У вас уже должно быть простое приложение из предыдущих шагов (например, файл app.py для FastAPI или index.js для Express).

Убедитесь, что ваше приложение слушает порт, который можно настроить через переменную окружения (это соответствует 3-му фактору, но нам это понадобится). Обычно приложения слушают 0.0.0.0, а не 127.0.0.1, чтобы быть доступными извне контейнера.

Шаг 2: Создание Dockerfile
В корневой папке вашего проекта создайте файл с именем Dockerfile (без расширения).

Шаг 3: Игнорирование лишнего (очень важно!)
Чтобы контейнер не копировал мусор с вашего компьютера (типа виртуального окружения venv или node_modules), создайте файл .dockerignore в той же папке.

Шаг 4: Сборка образа
Откройте терминал в папке с проектом и выполните команду сборки.

Шаг 5: Запуск контейнера
Запустите контейнер из созданного образа, пробросив порты.

Шаг 6: Проверка изоляции
Откройте браузер и перейдите по адресу http://localhost:8080. Вы должны увидеть ответ от приложения.

Попробуйте удалить Python (или Node.js) из вашей основной операционной системы (или просто временно переименуйте папку с интерпретатором).

Снова запустите контейнер командой.

Шаг 7 (Бонус): Чистота и повторяемость
Остановите контейнер (Ctrl+C или docker stop).

Удалите все зависимости из вашей основной системы (глобальные библиотеки).

Удалите папку venv или node_modules из папки проекта.

Пересоберите образ заново (docker build ...). Сборка пройдет успешно, так как установка зависимостей происходит внутри изолированной среды Docker, а не на вашей машине.


Практическая работа №3. Конфигурация

Цель: полностью отделить настройки приложения от кода. Научиться передавать конфигурацию через переменные окружения, чтобы один и тот же собранный артефакт мог работать в разных средах (dev/staging/prod) без пересборки.

Шаг 1: Выявление «зашитых» значений
Изучите код вашего приложения и найдите все строки, которые могут меняться от окружения к окружению: порт, строка подключения к базе данных, API‑ключи, хосты внешних сервисов. Составьте их список.

Шаг 2: Использование функции для получения значения переменной окружения
Для ключевых переменных предусмотрите разумные значения по умолчанию (например, port := os.Getenv("PORT"); if port == "" { port = "8080" }), чтобы приложение можно было запустить локально без задания всех переменных.

Шаг 3: Работа с секретами
Создайте файл .env.example со списком всех используемых переменных и примерами (не настоящими значениями). Добавьте .env в .gitignore, чтобы файл с реальными секретами никогда не попал в репозиторий. Секреты в коде должны жить только в окружении.

Шаг 4: Внедрение менеджера конфигурации
Подключите библиотеку spf13/viper (или аналогичную). Настройте так, чтобы читались переменные окружения, поддерживались значения по умолчанию и при необходимости мог читаться YAML‑файл для локальной разработки. Убедитесь, что переменные окружения имеют приоритет над файлом.

Шаг 5: Передача переменных в контейнер
Измените (обновите) команду запуска контейнера, чтобы передавать переменные окружения. Например, при использовании docker run добавьте флаги -e NAME=value. Если вы используете Docker Compose, пропишите переменные в секции environment вашего сервиса. Проверьте, что приложение в контейнере подхватывает новые значения.

Шаг 6: Проверка независимости
Запустите одно и то же приложение (один и тот же образ) сначала с одними переменными (например, PORT=5000), потом с другими. Убедитесь, что поведение меняется без пересборки.

Практическая работа №4. Подключаемые сервисы и отказ от состояния

Цель: научиться подключать внешние ресурсы (базы данных, кэши) как «прикрепляемые» через конфигурацию, а также сделать приложение полностью stateless – всё состояние должно храниться в этих внешних сервисах.

Шаг 1: Добавление базы данных через Docker Compose
Дополните ваш docker-compose.yml сервисом PostgreSQL (или MongoDB, Redis). Настройте сеть между контейнерами, чтобы ваше приложение могло обращаться к БД по имени сервиса.

Шаг 2: Подключение к БД через переменные окружения
Добавьте в код поддержку подключения к базе данных. Все параметры подключения (хост, порт, имя БД, пользователь, пароль) должны читаться из переменных окружения, как вы научились в практической работе №3. Убедитесь, что локально (без Docker) приложение может работать с лёгкой БД (например, SQLite), а в контейнере переключается на PostgreSQL – просто меняя переменные.

Шаг 3: Перенос состояния из памяти в БД
Найдите в приложении места, где данные хранятся в памяти процесса (например, пользовательские сессии, счётчики, временные файлы). Перенесите их в базу данных или во внешнее хранилище (Redis для сессий, S3‑совместимое хранилище для файлов).

Шаг 4: Симуляция сбоя процесса
Запустите несколько экземпляров приложения (например, docker-compose up --scale app=3). Остановите один из контейнеров с приложением (docker stop). Убедитесь, что другие экземпляры продолжают обрабатывать запросы, а данные (например, созданные пользователями записи) не пропали.

Шаг 5: Проверка stateless-приложения
Попробуйте полностью остановить все контейнеры с приложением, а затем запустить их заново. Все данные должны сохраниться в БД, а приложение – восстановить работу без потери функциональности.


Практическая работа №5. Сборка, релиз, запуск и масштабирование

Цель: освоить разделение этапов сборки и выполнения, а также обеспечить возможность горизонтального масштабирования простым добавлением процессов.

Шаг 1: Автоматизация сборки с уникальной меткой
Настройте простой CI‑пайплайн (например, GitHub Actions), который при каждом пуше в основную ветку собирает Docker‑образ и тегирует его уникальным идентификатором (номер сборки или commit hash). Образ пушится в registry (Docker Hub, GitHub Container Registry).

Шаг 2: Разделение сборки и конфигурации
Убедитесь, что собранный образ не содержит никаких переменных окружения. Все настройки для staging/prod (или другие аналогичные ветки) будут передаваться во время запуска. Это означает, что один и тот же образ может быть использован для разных окружений.

Шаг 3: Релиз как комбинация образа и конфигурация
При деплое на staging или production (или другие аналогичные ветки) CI-пайплайн берёт фиксированный образ (по тегу) и применяет к нему конфигурацию окружения (переменные). Релиз должен быть уникально идентифицируем (например, можно записывать в лог версию образа и окружение). Деплой не должен пересобирать образ.

Шаг 4: Масштабирование через увеличение числа процессов
Запустите несколько экземпляров вашего приложения (в Docker Compose укажите replicas или просто запустите несколько контейнеров вручную). Используйте инструмент нагрузочного тестирования, чтобы убедиться, что запросы распределяются между процессами.

Шаг 5: Отсутствие «липких сессий»
Убедитесь, что сессии приложения хранятся в централизованном хранилище (Redis), иначе запросы от одного пользователя могут попадать на разные экземпляры и терять сессию.

Практическая работа №6. Изоляция данных и логирование

Цель: организовать чёткий API для доступа к данным и перейти к структурному логированию в stdout как к потоку событий.

Шаг 1: Инкапсуляция данных за API
Проверьте, что внешние потребители (если бы они были) не имеют прямого доступа к базе данных вашего сервиса. Весь доступ должен осуществляться через публичные методы вашего приложения (REST, gRPC). При необходимости выделите слой сервисов или репозиториев, который будет реализовывать бизнес-логику и скрывать детали хранения.

Шаг 2: Внедрение структурного логирования
Замените все fmt.Println и log.Print на структурный логгер (пары ключ-значение). Настройте вывод в формате JSON. Каждая запись должна содержать как минимум: уровень (info/error), временную метку, имя сервиса и полезную информацию.

Шаг 3: Логирование в stdout
Убедитесь, что логи пишутся исключительно в стандартный вывод (stdout для обычных сообщений, stderr для ошибок). Никаких файловых логов. В контейнере Docker это позволит автоматически собирать логи через драйвер логирования.

Шаг 4: Сквозная трассировка (request ID)
Добавьте в каждый входящий HTTP-запрос генерацию уникального идентификатора (например, X-Request-ID). Пробрасывайте этот ID во все логи, связанные с обработкой запроса. Это позволит отслеживать путь запроса через систему.

Шаг 5: Просмотр логов в реальном времени
Запустите приложение и несколько раз обратитесь к нему. Наблюдайте поток логов в консоли (Docker автоматически выводит stdout). Если используете Docker Compose, можно посмотреть логи командой docker-compose logs -f.

Практическая работа №7. Равенство сред и корректное завершение

Цель: добиться максимальной близости окружений разработки и продакшна (финального продукта), а также научить приложение корректно обрабатывать сигналы завершения (graceful shutdown) и быстро запускаться.

Шаг 1: Унификация окружений
Убедитесь, что в разработке вы используете те же версии баз данных, очередей и других зависимостей, что и в продакшне. Все они должны быть описаны в docker-compose.yml и запускаться локально в контейнерах. Без «у меня на Mac стоит PostgreSQL 9.6, а на сервере 14» – используйте одинаковые образы.

Шаг 2: Быстрый старт
Оптимизируйте Dockerfile для минимизации времени запуска. Используйте легковесные базовые образы (alpine, scratch). Убедитесь, что приложение готово принимать трафик не позднее чем через пару секунд после старта контейнера.

Шаг 3: Сигналы завершения
Добавьте в код обработку сигналов SIGTERM и SIGINT. При получении сигнала приложение должно:

перестать принимать новые запросы;
дать завершиться уже выполняющимся запросам (в пределах тайм-аута);
закрыть все соединения с БД и другими сервисами;
только после этого завершить процесс.
Шаг 4: Тестирование
Запустите приложение и отправьте ему сигнал SIGTERM (например, docker kill --signal=SIGTERM <container>). Во время обработки сигнала попробуйте отправить несколько запросов. Убедитесь, что уже начавшиеся запросы успешно завершаются, а новые – получают корректный отказ (например, 503 Service Unavailable).

Шаг 5: Проверка «утилизируемости»
В облачных средах контейнеры могут быть остановлены в любой момент. Ваше приложение не должно терять данные и должно быстро перезапускаться. Под нагрузкой уничтожьте один контейнер и наблюдайте, как оркестратор (или вы вручную) запускает новый. Потери запросов быть не должно.


Практическая работа №8. Административные задачи и итоговая интеграция

Цель: научиться выполнять разовые административные задачи (миграции БД, консольные утилиты) как одноразовые процессы, не нарушая идемпотентность и неизменность инфраструктуры.

Шаг 1: Выделение административных команд
Определите типовые задачи, которые включают в себя применение миграций базы данных, создание административного пользователя, очистку кэша. Сделайте так, чтобы ваше приложение могло запускаться в разных режимах. Например, через флаги командной строки: ./app server – запуск основного сервиса, ./app migrate – выполнение миграций и выход.
Шаг 2: Миграции как одноразовый процесс
Реализуйте механизм миграций (можно использовать готовые библиотеки, например). Миграции должны применяться перед запуском новой версии приложения, но как отдельный шаг.

Шаг 3: Запуск одноразового контейнера
В CI/CD пайплайне перед деплоем новой версии добавьте шаг, который запускает одноразовый контейнер из того же образа, но с командой migrate. Контейнер должен использовать переменные окружения целевой среды. Если миграция завершилась ошибкой, деплой останавливается.

Шаг 4: Ручной запуск административных задач
Покажите, как можно вручную запустить, например, создание пользователя: docker run --rm -e ... myapp:latest create-admin --email=admin@example.com. При этом не требуется «заходить внутрь» работающего контейнера и что-то там менять вручную.

Шаг 5: Итоговая проверка
Продемонстрируйте полный цикл программы, соответствующий 12 факторам:

Изменение кода → пуск в репозиторий.
Автоматическая сборка нового образа.
Прогон миграций в staging (или аналог).
Деплой на staging (запуск контейнеров с новой версией).
Проверка работоспособности.
Откат на предыдущую версию заменой тега образа и перезапуском.
